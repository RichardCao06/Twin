第一部分 先读懂

1.1 是什么

DWS-Agent 是"我"在钉钉里的数字分身。 它学习我干过的活、写过的问题、正在开发的项目代码。当任何人在钉钉里问我、给我反馈问题时，它能像我一样及时回应：

- 能直接答的就答（基于我的知识，且标注"助理代答、待本人复核"）；

- 该立刻干的就指挥"项目外另一个真正能干活的工程 Agent"把活干完，并通知相关同事；

- 拿不准 / 要决策的就升级给我，把材料攒齐让我判断如何完成。

它把"我"蒸馏成一个 24 小时在线、答案能溯源、绝不乱承诺的"小号"。底层依托已安装的钉钉 CLI dws（v1.0.39，扫码登录），能读发消息、走审批、读写文档表格、管理待办与日程。

1.2 解决什么痛点

痛点（现状）DWS-Agent 带来的改变
我在忙 / 在睡，消息堆积，同事干等分身秒级先接，能答即答，不能答也先认领并给预期
同一个问题被反复问，我反复手敲分身从我的历史知识里直接调答案
小活也要我亲自上手分身把明确的小活派给"真正干活的工程 Agent"，干完回报
我答错 / 承诺过头要背锅分身明示"助理代答、待本人复核"，绝不冒充我做承诺
别人不知道事情进展到哪每条消息有状态、有回执、有溯源链

1.3 为什么难

  1. "代我说话"风险极高：说错一句话可能就是一次承诺、一次误导，且在信任与法律意义上"助理代我说的"往往等同于我说的。所以分身只能"代答 + 标注 + 可溯源"，绝不假装是我本人，且我可一键全局撤回并广播更正。
  2. "会思考的 AI"不能直接动手：能写会改的 AI 一旦手滑或被骗，删库、误发都是真实损失。所以我们把"动脑"和"动手"彻底拆开——思考的只出"意图 JSON"，动手的是一个不带 AI、只认规则的执行官。
  3. AI 会"自信地编造"：所以铁律是"不确定就升级，绝不编"，并且答案必须能指回出处。
  4. 干活不能信 AI 自己说的"我干完了"：必须有独立验收；谁干谁验收会作弊。
  5. 最隐蔽的根问题——语义判定 vs 无 LLM 执行：判断一句话是否"含承诺"、一段 diff 是否"高危"，本质都是语义任务，纯规则做不全。这是架构层的根本张力，解决方式见 1.4 第 8 条 与 3.1。

1.4 核心思想（七条不可推翻的硬约束）

#硬约束大白话
1思考与执行分离会思考的 AI（分诊 Worker、干活 Agent）永不亲手做写操作，只产意图 JSON；真正执行的是无 LLM、按规则判级的确定性 Executor，靠 OS 权限 + token 隔离（非 PATH）。
2代答身份明示"助理代答 / 待本人复核"，绝不冒充本人做承诺。
3干活自动化上限 W2最多到"提 PR / 提交草稿"；合并 / 部署 / 发布（W3）永久人确认。
4不确定就升级（abstain），绝不编代答必可溯源；干活必独立验收（不信自评）。
5认证隔离本机单实例 + 共享 ~/.dws 凭证 + 刷新文件锁；auth export/import/logout/reset 列 never。
6三套并列风险分级R0–R3（dws 调用）、C0–C2/abstain（代答）、W0–W3（干活），各有永久人确认终态。绝无"全自动"终态。
7数据治理落盘 Keychain 加密、字段最小化、分级脱敏、污点传播、对外答问出口管控。

补充原则（吸收对抗审查 C1）：判级分两类。 结构性判级（动作类型、目标分支、文件路径、API 名）用确定性规则、放在 Executor；语义性判级（是否含承诺、是否敏感、答案是否真的由证据支撑）由思考侧模型产出并默认保守降级，Executor 只消费这些"已标注的保守标签"做 AND 闸门。无 LLM 组件绝不被强加语义判断的全部责任。

第二部分 三个核心子系统

2.1 知识蒸馏层 KDL

2.1.1 定位与边界

KDL 是数字分身的"长期记忆 + 可代答知识库"，只做三件事：摄取 → 蒸馏 → 供检索。它不产生对外答复（那是分诊 Worker 的事），只向 Worker 提供"带证据、带权威级、带新鲜度"的候选知识片段。

硬约束映射：自身无 LLM 写执行权（蒸馏中 LLM 只产"知识候选 JSON"，落库由无 LLM 的 Ingestor 按规则做）；代答用片段必可溯源；"升为权威知识"必须经本人确认这一确定性闸门；入库即脱敏 + 污点传播 + 出口管控。

2.1.2 四类知识源与统一知识单元 KU

源代号原始形态蒸馏产物新鲜度风险
在开发项目代码CODEcommit、函数/类/接口、README、配置代码事实卡（symbol→行为/契约/坑）极高（代码漂移）
历史 issue / 写过的问题ISSUEissue 正文、标题、标签、状态流转症状→根因→处置 卡中
历史问答对QA钉钉消息、文档评论、邮件往返规范化 Q→A 对中
干过的工作套路PLAYBOOK重复操作序列、审批/部署流程SOP/Runbook 步骤卡低-中

KU 数据模型（字段最小化）：

KU {
  ku_id, source_type(CODE|ISSUE|QA|PLAYBOOK), title, body, body_redacted(bool),
  taint(CLEAN|INTERNAL|SENSITIVE),
  authority(DRAFT|REVIEWED|AUTHORITATIVE|DEPRECATED),
  public_ok(bool),          -- 是否预先核定"可对外公开"（吸收审查 A1）
  confidence(0..1),         -- 仅排序参考，不决定能否答
  freshness(FRESH|STALE|EXPIRED|UNKNOWN),
  provenance[],             -- 溯源，缺失则不可代答
  repo, commit_sha, file_path, symbol, line_range, content_hash,  -- 仅 CODE
  created_at, updated_at, last_verified_at, expires_at, superseded_by, owner
}

Provenance {
  kind(COMMIT|ISSUE_URL|MSG_ID|DOC_ID|MAIL_ID|FILE),
  ref, quote(脱敏后摘录), quote_taint, captured_at, retrievable(bool)
}

规则： provenance 为空 → authority 强制 DRAFT 且不得用于代答（对应"代答必可溯源"）。

2.1.3 摄取与蒸馏流水线

采集 Collector → 脱敏 Redactor → 蒸馏 Distiller(LLM,只产JSON)
   → 校验 Validator(无LLM) → 入库 Ingestor(无LLM,确定性, authority=DRAFT)
   → 索引 Indexer(FTS+符号+按需向量)

关键：Distiller 是唯一用 LLM 的环节，输出严格 JSON、无任何副作用；Validator/Ingestor/Indexer 全无 LLM、纯规则。

各源蒸馏要点：

  • CODE（抗漂移是核心）：单元 = 符号级（函数/类/接口/导出 API），强制绑定 commit_sha + file_path + symbol + line_range + content_hash。蒸馏三类内容：①契约 ②行为/副作用 ③已知坑。禁止凭空概括：映射不到具体符号即丢弃。
  • ISSUE：蒸馏为 症状→根因→解决→关联代码符号，与 CODE-KU 双向引用，代码改动可连带标 stale。
  • QA（自动配对，吸收审查 A5）：配对器用规则 + 轻量 LLM 判定（不落库），信号 = 同线程疑问句/@我 + 我的后续回复、reply/quote、时间邻近、邮件 In-Reply-To。强校验回复确实出自本人 account 且无中间插话，防抢答污染。一律入库 DRAFT。
  • PLAYBOOK：归纳触发条件→步骤→校验点→责任人。是干活 Agent 的意图模板来源，但本身 DRAFT，须本人确认才可被自动援引为 W1+ 依据。

2.1.4 代码漂移与新鲜度治理（重点）

freshness含义判定Worker 可用性
FRESH绑定 commit 仍是最新、hash 匹配校验通过可作依据答（按权威级）
STALE符号在 HEAD 已变但仍存在hash 漂移仅"参考性"答，必降级 + 提示"可能已过期，正在复核"
EXPIRED符号已删/文件不存/源不可取回找不到符号禁止代答，转 abstain/升级
UNKNOWN未绑定或从未校验无 commit 绑定视同 STALE 下限

校验触发三层： ①事件触发（监听 push/merge，对受影响 file_path 反查 KU 逐符号重算 hash）；②定时巡检（超阈值未验证批量复检）；③检索时懒校验（最后防线）：Worker 取用 CODE-KU 时实时轻量 hash 比对再返回。

吸收审查 D1（TTL/TOCTOU）： 懒校验的秒级缓存对"登录接口 500"这类高频变更场景仍可能失效。修复：高敏/高频域不用缓存；对外答复附"基于 commit X"的免责，并在发出瞬间做二次校验，窗口内代码再变即拦截。

漂移联动失效（污点式失效传播）： 符号变更 → 直接绑定的 CODE-KU 置 STALE → 沿 ISSUE↔CODE、QA↔CODE 引用边把引用该符号的派生 KU 标 derived_stale。AUTHORITATIVE 一旦底层代码漂移 → 自动降级为 REVIEWED + STALE 并生成"待本人复核"任务。权威性不能跨过代码变更自动延续。

AUTHORITATIVE仅本人强确认可较直接代答（仍标助理身份）C0/C1
DEPRECATED本人/系统不可用不可用

升级动作由本人在钉钉对"复核卡"点确认完成，事件写审计；Executor 按规则改 authority 字段，无 LLM 参与。

2.1.6 可溯源、abstain 与检索

可溯源（硬约束）： 每条 KU ≥1 条 provenance，否则锁死 DRAFT 且 serve_blocked。断链检测：定期校验 retrievable，来源被删/不可达 → 置 EXPIRED/evidence_broken，禁止代答。吸收审查 A7： quote 也必须过脱敏与污点档；对外只给"出处标识"，不直接展示原文 quote。

Abstain（不确定就升级，绝不编）： Serve 层返回结构化裁决 Verdict{decision, kus, reason}。强制 ABSTAIN 条件（任一命中）：

  1. 无 KU 命中 / 命中全为 DRAFT；
  2. 命中全 EXPIRED，或最佳为 STALE 且无 FRESH 替代；
  3. 顶级候选 provenance 断链；
  4. 候选互相矛盾（仅 REVIEWED+ 才算冲突源，DRAFT 不参与对立面——吸收审查 A5 防升级 DoS）；
  5. 涉及承诺/决策/对外口径；
  6. 检索 top1 相关度双低（语义 + 词面）。

吸收审查 A4（grounding gap）： 检索回查只保证"引用存在"，不保证"答案=引用"。新增无 LLM 的"答案-证据蕴含校验"（NLI/字符串覆盖率门）：覆盖率不足强制 abstain；abstain 判定与拟答用不同模型/不同 prompt，降低共模注入。

检索（FTS + 符号索引，按需向量）：

层技术用途启用
L1 符号索引倒排 symbol/file/repo→ku_id代码精确问题始终（CODE 优先）
L2 FTSSQLite FTS5/关键词词面匹配始终
L3 向量句向量 + ANN语义近似按需：L1/L2 不足时

融合排序 score = w_f·freshness + w_a·authority + w_r·相似度；EXPIRED/DEPRECATED 直接出局，STALE 强降权。freshness/authority 是硬门槛，相似度只在通过门槛者间排序。吸收审查 C3：SENSITIVE 内容不进入面向外部提问的同一向量空间（分库），避免被语义检索召回到对外代答上下文。

2.3 双 Agent 编排（指挥干活 Agent）

2.3.1 核心立场：干活 Agent 是不可信外包

干活 Agent（下称 WA）是项目外、能写代码/动文件/能提交的强力执行体。根本假设：WA 可能能力很强，但永远不可信——可能误解需求、越权、幻觉式"已完成"、自评注水、被污染上下文劫持。三道闸门夹住它：

我(本人) ──升级决策──┐
分诊Worker ──产意图──> 确定性Executor(无LLM,判级,持token)
                     │  TaskSpec(脱敏/边界)        │ W0-W3 落地
                     ▼                            ▼
              ┌─────────────┐          ┌──────────────────────┐
              │ Worker-Agent│ 产diff/  │ 独立用户+容器沙箱      │
              │ (不可信外包) │ PR/声明  │ +egress审计+token隔离 │
              └─────────────┘          └──────────────────────┘
                     │ 自评(零信任)
                     ▼
        独立验收器(干净沙箱重跑测试/CI/diff静态核验,不读WA自评)
                     │ 真实结果
                     ▼
        闭环回报 → 助理署名R3对外发送 / 不过→转人工
  • 入口闸 TaskSpec：只给脱敏、有边界的任务，不给原始凭证/客户数据。
  • 执行闸 W0–W3 + 确定性 Executor：才能落地写操作。
  • 出口闸 独立验收 + 出口管控：WA 对外发声权为零。

2.3.2 移交协议 TaskSpec（入口闸）

字段必填含义不可信防护点
task_id✓全局唯一，贯穿验收/回报/审计防混淆/重放，产物按 task_id 绑定
goal✓一句话目标只描述目标，不含商业背景
context✓已脱敏的必要上下文客户名→占位、密钥剔除、内部 URL→代号；污点字段不得进入
acceptance✓机器可判的验收标准WA 看得到标准，判定权不在 WA；标准本身须确定性派生或本人核定（防注入，见下）
risk_bound✓本任务最高动作级 W0/W1/W2（W3 永不签发给 WA）Executor 对照拒绝越级
scope✓允许触达的路径/仓库/分支白名单越界写=硬拒绝
deadline✓截止时间；超时自动转人工防无限挂起占资源
forbidden✓黑名单：禁改文件、禁调外网域、禁碰命令默认含 CI/workflow/依赖锁文件（见下）
secrets_ref○仅必要时给"token 句柄"而非明文凭证永不进 WA 上下文，且句柄绑定动作模板 + 配额

脱敏管线（context 生成强制）：分级脱敏（R0 原文 / R1 代号 / R2 占位符摘要 / R3 禁出→改升级）；污点传播继承最高级、不可洗白；签发前 Executor 正则 + 实体扫描，命中密钥/邮箱/客户实体即阻断签发并升级。

2.3.3 干活动作风险分级 W0–W3（执行闸）

级别名称典型动作落地者需人确认隔离手段
W0只读探查读代码/测试/日志、git logExecutor 只读 token否只读挂载，沙箱无写权
W1本地改+测试改代码、跑单测、构建、lintExecutor 写入 worktree 沙箱否（产物隔离）独立用户 + 容器 + worktree + egress 全代理审计
W2提 PR/草稿push feature 分支、开 draft PR、提文档草稿Executor 受限 push token否（自动上限）PR 必 draft；不能 target 受保护分支
W3合并/部署/发布merge、deploy、release、改生产/权限、对外正式发文仅本人经 R2/R3 确认是（永久，绝无例外）WA 与 Executor 均无 W3 token；人在环

Executor 裁决（无 LLM，纯规则）：动作级 > risk_bound 硬拒；触达 scope 外或命中 forbidden 硬拒；W1 全程在沙箱、主工作区零污染；W2 强制 draft=true 且 base ∈ 非受保护分支；任何"想跨 W3"的意图一律转升级。

吸收审查 B1（PR 本身即攻击载体）： draft PR 可在 CI/Actions 植入恶意 workflow（pull_request_target 类漏洞窃取 secrets）、改依赖清单引供应链执行、用描述社工诱导合并。修复：forbidden 默认含 CI/workflow/依赖锁文件，改动这些强制人工；PR target 必须是无 CI 自动触发的隔离分支。

吸收审查 B3（沙箱逃逸/外带）： 沙箱必须是独立用户 + 容器，与主机不共享 ~/.dws（不同 uid + 文件权限 + 不挂载）；egress 完全代理审计——连包镜像也走审计代理记录全部出站（防 DNS 隧道/把数据编码进包请求外带）。

吸收审查 B6（句柄间接滥用）： secrets_ref 句柄须绑定到具体动作模板与配额，Executor 对句柄用途做规则白名单，而非"WA 说用就用"，防 WA 借句柄在 scope 内做有害事（刷垃圾 push、耗 API 额度）。

2.3.4 独立验收（不信自评）

铁律：验收器从不读取 WA 的"我已完成/测试通过"自评文本，只看可独立复现的客观证据。 验收器是无 LLM 的确定性流水线。

维度方法通过条件反作弊
功能在干净、无 secrets、无网、用后即焚沙箱重跑 acceptance.tests全绿、指定 -count 稳定重新 checkout WA 分支自己跑，忽略 WA 日志
回归跑全量既有测试无新增失败对比 base 基线
CI读 CI 真值（如 required）CI 绿读 CI 系统真值，非 WA 转述
Diff 静态核验解析 PR diff满足 diff_constraints（文件数/路径/无禁改/无新增密钥）静态扫描密钥与 forbidden
边界合规比对 scope/forbidden零越界命中即否决
无副作用沙箱网络/文件审计日志无未授权外联/禁改eBPF/审计日志为准

判定与转人工：全 PASS→闭环回报；不过→自动重试 ≤N 次（失败证据经脱敏后塞回 context，仅"断言摘要"非原始日志——吸收审查 B4）；越界/命中 forbidden/检出密钥/试图 W3/超 deadline → 直接转人工不重试。重试不得放宽 acceptance（放宽属决策，只能本人做）。 "自评通过但独立验收不过"记为 WA 可信度负分，后续同类任务自动收紧 risk_bound。

吸收审查 B2（验收标准被注入污染）： acceptance 写在 TaskSpec、由分诊 Worker（LLM）生成，攻击者可经 bug 反馈注入，诱导生成"跑一个必过空测试"的验收门。修复：acceptance 必须由确定性规则从需求派生或由本人核定，LLM 不得自由撰写验收门；高风险任务的 acceptance 须人确认。

吸收审查 B5（人被诱导确认 W3）： 带外签名只保证"是我点的"，不保证"我看懂了"。攻击者用伪造 U3 制造时间压力 + 看似无害的 PR + WA 影响的验收摘要话术。修复：确认卡必须展示确定性生成、WA 不可影响的 diff 摘要与风险高亮（改了哪些敏感文件、是否动 CI/依赖/权限）；高风险 PR 引入冷静期或第二人复核。

3.1 PolicyGate：三套分级统一裁决

PolicyGate 是唯一把三套分级合并裁决并做对外出口脱敏的确定性组件。它不思考、不调 LLM，只对输入意图 JSON 做规则匹配、查表、字段扫描、签名校验。

分诊Worker / 干活Agent ──意图JSON──► ┌──────────────────────────┐
                                     │      PolicyGate(无LLM)     │ ← 唯一收口
                                     │  统一裁决 R×C×W + 出口管控  │
                                     └──────────────────────────┘
                  ┌──────────┬──────────────┬─────────────┐
                  ▼          ▼              ▼             ▼
              自动放行    暂存/草稿      升级给本人      拒绝/熔断

三维取最严（短板原则）： 每条意图同时打 (R, C, W) 标签，最终动作 = 三维各自允许动作的最小交集。

decision = MIN_BY_STRICTNESS( resolve_R(...), resolve_C(...), resolve_W(...) )
严格度序：AUTO < DRAFT < HUMAN_CONFIRM < DENY

裁决矩阵（代表性组合）：

场景RCW合并结果
群里问功能怎么用，答案来自公开文档R2C0-DRAFT→过出口管控后自动发
同上但涉及未发布计划R2abstain-DENY 代答正文 + 升级
反馈 bug，Agent 定位改好提 PRR1C1W2W2 自动提 PR + C1 待复核回复

额外门槛：出口管控优先级高于置信度。 代答即使 C0，只要触达 R2（对外群发），必须先过出口管控（3.2）再放行；任一红线命中即降级，绝不因 C0 豁免。

永久人确认终态（不可被任何自动路径覆盖）： R3 全部、W3 全部、C 维度对外承诺性回复、出口管控命中"密钥/内部代码/未公开计划"、dws never 项。

吸收审查 C1（根问题）： PolicyGate 只做结构性判级（动作类型、目标分支、文件路径、API 名）；语义性判级（承诺、敏感、grounding）放在思考侧、默认保守降级，PolicyGate 只消费已标注的保守标签做 AND。两类判级 AND 后取严，确保无 LLM 组件不独自承担语义判断。

3.2 对外答问出口管控

代答正文落地前的最后一道确定性闸，独立于置信度。三步：身份定级 → 颗粒度裁剪 → 红线扫描。

身份/群信任档：

信任档典型来源可答颗粒度上限
T0 外部/未知/外部群非本企业、客户群仅公开口径
T1 内部普通同事本企业、非项目组+ 一般进展、非敏感答疑
T2 项目组/直接协作方同项目成员+ 项目内进展、接口约定（不含密钥/未公开计划）
T3 本人/受托代理仅本人带外身份全量（写操作仍受 R/W 约束）

群 > 个人取更严，且以"信息会被谁看到"为准（吸收审查 A2）： 外部群里即便对方是内部同事，按 T0 群上限裁剪。进一步修复身份假冒与转发外流：

- 新成员入群触发历史敏感消息撤回评估；

3.3 Kill Switch 与带外确认通道

Kill Switch（熔断）： 两种触发，效果都是 PolicyGate 立即把非 R0 动作降为 DENY，只留只读与升级。

类型触发行为
前置硬确认带外 CLI dws-gate kill --scope=all\|answer\|work写 ~/.dws/gate.lock（Keychain 加密），每次裁决前先读锁
频次熔断（自动）单位时间动作超阈值（5min 内 R2+ > N、同群代答 > M、W2 提 PR > K）自动置锁 + 升级，需带外解锁

吸收审查 D2： 熔断分级——区分降级到只读 vs 全停，区分"攻击触发"与"自身故障"；阈值按域而非全局，防攻击者主动触发熔断瘫痪分身（用熔断本身做 DoS）。

带外确认通道（绝不解析群文本）：核心原则——确认信号永不来自钉钉消息体，防社工/注入。

1. PolicyGate 产待确认项 → 写 pending 队列，生成 nonce(一次性、随机、绑意图哈希)
2. 经 DING/本人专属私聊 仅推送「确认链接指向本地CLI + nonce摘要」(通知,非授权入口)
3. 本人本机执行：dws-gate approve --nonce=<n> --sig
   本地CLI 用本机 Keychain 私钥对 nonce+intent_hash 签名
4. PolicyGate 校验签名 + nonce未过期未用过 + 意图哈希一致 → 解冻 → 交 Executor 执行

nonce 一次性、TTL 短（约 10 分钟）、与意图哈希强绑定，改一字节即失效。群里即使原样贴出 nonce 也无法伪造签名。kill/unlock/R3/W3/never 项只能走此带外签名通道。
