## 三、改进措施（防复发）

| 针对 | 改进（可立即执行） |
|---|---|
| 截断 id 虚构 | **关键标识符（会话/PR/任务 id）显示可截断，查询/操作必须用完整值**；读到对话先核对 `single_chat` + 参与者再下结论 |
| 凭推断下结论 | **「运行时/部署/归属」类判断，先实测（curl/kubectl/API）再说**；代码只说明"设计意图"，不替代"实际行为" |
| 改共享组件没枚举依赖方 | **改 SSO/Redis/网关/共享配置前，先枚举所有使用方**（#24 改 is-concurrent 前若枚举出 backend/editor/square 共用，互踢当场可见） |
| 操作 PR/issue 凭假设 | 动手前先查状态（`gh pr view` 一条命令），别假设已合/未合 |
| PR 英文 / 漏关联 | **task 模板固化**：PR 用中文、关联 issue（`Closes`/回写）、必带验证步骤 |
| 脚本白跑 | 长轮询/脚本先小样验证有效再放后台 |

## 四、可进一步自动化的环节（按价值排序）

1. **改共享组件的「影响面自动体检」** ⭐ 最该做——改 SSO/网关/共享 Redis 时，自动列出所有依赖方 + 跑互踢/冲突检测。**今天的互踢本应在改 #24 时被自动拦下**，而非上线后实测发现。
2. **诊断 playbook 自动化**——登录失败 / 404 / CI 超时 / 服务不可用 等高频问题，固化成"一键诊断 task"（自动查 pod 状态、日志、DNS、网关路由、会话），免每次手摸。
3. **issue 自动分诊**——轮询 `feedback` 新 issue，按 `line/type` label 自动 KDL 排查 + 建 **draft**（人确认 publish），把手动前半段自动化。
4. **PR 出来自动审查 + 回写**——task 出 PR → 自动代码审查 + 影响面分析 + 回写原 issue，闭环不靠人催。
5. **UAT 环境清单进 KDL**——诊断时自动检索"去哪查日志/连哪个库"，不用每次重新摸 namespace。
6. **CI 健康哨兵**——超时/失败自动给根因提示（如"多架构 QEMU 导致超时"）。

## 五、最优先

今天最大的风险不是技术难题，而是**"自信地说错"**（虚构讨论、审错 PR）。所以第 1 优先：

- 给执行立"**先实测再结论**"的硬规矩；
- 把「**影响面自动体检**」做出来——让系统在拍脑袋时兜底（这正是 #24 互踢应被提前发现的机制）。
