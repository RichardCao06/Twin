# 第一部分 先读懂

## 1.1 是什么

> **DWS-Agent 是一个个人 + 项目的工程助理平台**，由三根柱子撑起来：
> - **知识（Knowledge）** —— KDL 灌入 Workspace 所有仓库 + 钉钉文档形成 7289+ KU，叠加 `MEMORY.md` 长期记忆和 `docs/retro/` 复盘沉淀，作为决策、答复、派活的事实来源。
> - **编排（Orchestration）** —— 会思考的（Claude、subagent、远程 Worker）只产意图；动手的由不带 LLM 的确定性 Executor + PolicyGate 落地；Human-in-loop 兜底所有对外可见的动作。
> - **复盘（Retrospective）** —— 每次事故 / 上线 / 踩坑写复盘 → 提炼进 `MEMORY.md` → 反过来指导下次决策。这个循环是能力持续进化的机制，而不是可选动作。

**钉钉（`dws` CLI，v1.0.39）是这个平台的入口通道之一**——最早的触发场景，仍在跑 feedback 巡检和 MVP1 代答链路。但**它不再是主战场**：日常真正在跑的是跨仓库工程发布 / uat 部署 / 生产验证 / 复盘反哺，钉钉只在其中承担"消息进"（读群里问题、feedback 通知）和"消息出"（代答、告警）两条通道。

底层能力清单：钉钉侧依托 `dws` CLI 能读发消息 / 走审批 / 读写文档表格 / 管待办日程；平台侧依托 ClaudeCenter Worker 桥接、本地 subagent（Explore/Plan/uat-deploy/uat-verify/prod-verify）、KubeSphere 日志只读、生产 DB 直连等能力，覆盖真实的工程日常。

## 1.2 解决什么痛点

**日常工程侧（当前主要战场）：**

| 痛点（现状） | 平台带来的改变 |
|---|---|
| 跨仓库排查（Java/Vue/Python 多语言、多项目），上下文找不齐 | KDL 覆盖 Workspace 全部代码 + 文档，一次检索定位相关文件与前例 |
| 派活给 Worker/Copilot 时，需要重复交代验收标准、"不做清单"、参照物 | `dws-agent task create` 把带完整验收标准的 draft 落成文件，`publish` 后 Worker 认领执行 |
| 生产 bug 排查要来回切 pod / 拷贝 Token / 拼命令 | `ks_logs.py` 一行查生产日志，`prod.env` psql 直连 DB，都进入版本可复用 |
| Uat 部署 CI 不稳，回归验证靠人肉 | `uat-deploy` 本地 build+push 兜底，`uat-verify` Playwright 走 E2E 出报告 |
| 同类事故反复踩（长期分支 vs main、静默 catch、版本不存在导出空包...） | 每次复盘沉淀成 `MEMORY.md` 条目，下次决策时自动加载 |

**钉钉侧（原始入口，仍在跑）：**

| 痛点（现状） | 分身带来的改变 |
|---|---|
| 我在忙 / 在睡，消息堆积，同事干等 | 分身秒级先接，能答即答，不能答也先认领并给预期 |
| 同一个问题被反复问，我反复手敲 | 分身从我的历史知识里直接调答案 |
| 小活也要我亲自上手 | 分身把明确的小活派给"真正干活的工程 Agent"，干完回报 |
| 我答错 / 承诺过头要背锅 | 分身明示"助理代答、待本人复核"，绝不冒充我做承诺 |
| 别人不知道事情进展到哪 | 每条消息有状态、有回执、有溯源链 |

## 1.3 为什么难

1. **"代我说话"风险极高**：说错一句话可能就是一次承诺、一次误导，且在信任与法律意义上"助理代我说的"往往等同于我说的。所以分身只能"代答 + 标注 + 可溯源"，绝不假装是我本人，且我可一键全局撤回并广播更正。
2. **"会思考的 AI"不能直接动手**：能写会改的 AI 一旦手滑或被骗，删库、误发都是真实损失。所以我们把"动脑"和"动手"彻底拆开——思考的只出"意图 JSON"，动手的是一个**不带 AI、只认规则**的执行官。
3. **AI 会"自信地编造"**：所以铁律是"不确定就升级，绝不编"，并且答案必须能指回出处。
4. **干活不能信 AI 自己说的"我干完了"**：必须有独立验收；谁干谁验收会作弊。
5. **最隐蔽的根问题——语义判定 vs 无 LLM 执行**：判断一句话是否"含承诺"、一段 diff 是否"高危"，本质都是语义任务，纯规则做不全。这是架构层的根本张力，解决方式见 [1.4 第 8 条](#14-核心思想七条不可推翻的硬约束) 与 [3.1](#31-policygate三套分级统一裁决)。

## 1.4 核心思想（七条不可推翻的硬约束）

| # | 硬约束 | 大白话 |
|---|---|---|
| 1 | **思考与执行分离** | 会思考的 AI（分诊 Worker、干活 Agent）永不亲手做写操作，只产意图 JSON；真正执行的是无 LLM、按规则判级的确定性 Executor，靠 OS 权限 + token 隔离（非 PATH）。 |
| 2 | **代答身份** | 明示"助理代答 / 待本人复核"，绝不冒充本人做承诺。 |
| 3 | **干活自动化上限 W2** | 最多到"提 PR / 提交草稿"；合并 / 部署 / 发布（W3）永久人确认。 |
| 4 | **不确定就升级（abstain），绝不编** | 代答必可溯源；干活必独立验收（不信自评）。 |
| 5 | **认证隔离** | 本机单实例 + 共享 `~/.dws` 凭证 + 刷新文件锁；`auth export/import/logout/reset` 列 never。 |
| 6 | **三套并列风险分级** | R0–R3（dws 调用）、C0–C2/abstain（代答）、W0–W3（干活），各有永久人确认终态。**绝无"全自动"终态。** |
| 7 | **数据治理** | 落盘 Keychain 加密、字段最小化、分级脱敏、污点传播、对外答问出口管控。 |

> **补充原则（吸收对抗审查 C1）：判级分两类。** 结构性判级（动作类型、目标分支、文件路径、API 名）用确定性规则、放在 Executor；语义性判级（是否含承诺、是否敏感、答案是否真的由证据支撑）由思考侧模型产出并**默认保守降级**，Executor 只消费这些"已标注的保守标签"做 AND 闸门。无 LLM 组件绝不被强加语义判断的全部责任。

## 1.5 四个角色与信任边界

把整个系统想象成一家小型"代办事务所"：

| 角色 | 通俗比喻 | 能干什么 | **绝不能**干什么 | 含 LLM |
|---|---|---|---|---|
| **① 知识层 KB** | 记忆库 + 档案室 | 存历史问答、文档、项目代码、工单；按问题检索并给"带出处"的素材 | 不下结论、不发消息、不动外部系统 | 否（检索/向量，无生成） |
| **② 分诊 Agent** | 前台 + 分诊医生 | 读消息、调知识层、判"能答/派活/升级/归档"、起草代答稿与派活意图 | 不亲手发消息、不亲手改代码、不冒充本人 | 是 |
| **③ 干活 Agent** | 外包工程师（不可信） | 接派活意图后写代码/改文档/出 PR 草稿，产"工作意图" | 不自己合并/部署/发布、不自评通过、不直接对外回话 | 是 |
| **④ 执行官 Executor** | 出纳/盖章员（照章办事） | 接意图 JSON，按 R/C/W 规则判级、靠 OS 权限 + token 真正执行 dws 写操作 | 不思考、不改写意图、不超规则放行；遇终态必停等人 | **否（确定性规则机）** |

**信任边界（最关键的一张图，文字版）：**

```
        会思考（可能犯错/可能被骗） │ 不思考（确定、可审计）
  ─────────────────────────────────┼──────────────────────────────
   ② 分诊Agent  ③ 干活Agent         │       ④ 执行官 Executor
        │            │              │              │
   只产"意图JSON" 只产"意图JSON"     │   只认意图JSON + 规则 + token
        └─────┬──────┘              │              │
              ▼                     │              ▼
        ╔═══════════ 信任边界（不可逾越）═══════════╗
        ║ 思考侧永远拿不到执行 token；执行侧永远不   ║
        ║ 调用 LLM。两侧只通过"意图JSON"单向通信。   ║
        ╚════════════════════════════════════════════╝
```

- **思考侧（②③）零执行权**：它们手里没有任何能真正发消息 / 改库的钥匙，只能写"我想这么干"的纸条（意图 JSON）。
- **执行侧（④）零思考权**：它不调用任何大模型，不会被"提示词注入"骗，只按写死的规则判级、按 token 隔离的权限干活，碰到永久人确认终态就停。
- **token 隔离靠 OS 权限，不靠 PATH**：执行能力来自操作系统层面的文件权限与 token 持有（不同 uid + 文件权限），而非"把命令放进 PATH"这种可被绕过的方式。

## 1.6 三套并列风险分级

任何一步都同时受三套分级约束，**每套都有"永久人确认"终态，绝无全自动终态**：

| 维度 | 含义（大白话） | 分级 | 永久人确认终态 |
|---|---|---|---|
| **R0–R3** | 这次 dws 调用本身有多危险 | R0 只读 → R1 低危写 → R2 对外可见写 → R3 高危/不可逆/治理动作 | R3 必须人确认 |
| **C0–C2 / abstain** | 代我说话有多大把握 | C0 直接答 / C1 答 + 标注待复核 / C2 仅给草稿待我发 / **abstain 不确定就升级绝不编** | C2 与 abstain 入升级 |
| **W0–W3** | 替我动手到哪一步 | W0 只读分析 / W1 沙箱内改 / W2 提 PR 或草稿（自动上限）/ **W3 合并·部署·发布** | W3 永久人确认 |

## 1.7 端到端流程：一条钉钉消息进来

**总览（七步）：**

```
钉钉消息
   │
[1] 接入与去敏 ──► [2] 分诊Agent理解意图
                          │
                  [3] 知识层检索（带出处）
                          │
                  [4] 路由四选一
        ┌───────────┬───────────┬───────────┬──────────┐
        ▼           ▼           ▼           ▼
     A 代答      B 派活      C 升级我     D 归档
        │           │           │           │
   [5]出口管控发出  干活Agent     攒材料+证据   记录入库
        │          出W意图        给我决策     （无对外）
   [6]对外回话    [6]执行官≤W2落地
        │           │
   [7]回执+溯源   [7]独立验收→通知同事→回报
```

| 步骤 | 谁来做 | 关键动作 | 分级落点 |
|---|---|---|---|
| **[1] 接入与去敏** | 执行官（无 LLM） | dws 监听消息；落盘前字段最小化 + 分级脱敏 + 打"污点标记"，污点沿全流程传播 | R0 |
| **[2] 理解意图** | 分诊 Agent | 判断是提问/报 bug/提需求/闲聊，涉及哪个项目、紧急度，抽"问题指纹" | 只读思考 |
| **[3] 知识检索** | 知识层（纯检索无生成） | 召回 Top-N 片段，每条带出处；召回强且一致→倾向 C0/C1，弱或矛盾→倾向 abstain | R0 |
| **[4] 路由四选一** | 分诊判定 / 执行官执行 | A 代答 / B 派活 / C 升级我 / D 归档 | C/W + R |
| **[5] 执行** | 执行官 / 干活 Agent | A 走出口管控（加标识、查污点、查承诺）；B 干活 Agent 最多到 W2 | C+R / W+R |
| **[6] 对外/落地** | 执行官 | A 发出代答（R 写）；B 落地 W≤2 产物 | R1/R2 / W≤2 |
| **[7] 验收回报** | 独立验收器 + 执行官 | B 必独立验收（不信自评）；通过→通知同事 + 回报我；不通过→退回/升级。全程留溯源链 | — |

**路由四选一详表：**

| 路 | 触发条件 | 谁做 | 分级落点 |
|---|---|---|---|
| **A 代答** | 知识层召回充分、可溯源、不含对外承诺 | 分诊起草 → 执行官发出 | C0/C1/C2 + R1/R2 |
| **B 派活** | 明确、可独立验收的工程任务 | 干活 Agent 产 W 意图 → 执行官落地 | W0–W2 + 对应 R |
| **C 升级我** | abstain / 要决策 / 涉对外承诺 / 触 R3 或 W3 | 分诊攒材料 → DING 给我 | C-abstain，待人 |
| **D 归档** | 闲聊、已自动解决、无需回应 | 执行官静默入库 | R0 |

**一个具体例子（让任何人秒懂，且已修正对抗审查 D3）：**

> 同事钉钉发：「登录接口又 500 了，急！」
> 1. 接入去敏 → 2. 分诊判定：报 bug、高紧急、属"登录项目"。
> 3. 知识层召回：上周同款工单 + 报错堆栈出处。
> 4. 路由 = **B 派活**（明确、可验收）。**注意**：诊断所需的生产日志/堆栈可能含 PII/密钥（高污点），故进入干活 Agent 前必经脱敏；若诊断必须读高敏生产数据，则改为**升级给我**而非自动派活。
> 5–6. 干活 Agent 在隔离沙箱定位空指针，写修复**提 draft PR（W2 封顶，不自动合并）**。
> 7. 独立验收器在干净、无凭证、用后即焚的沙箱跑回归测试 → 通过后执行官以"助理代答/待本人复核"身份回同事「已定位并提交修复 PR，待 XX 复核合并」，同时 DING 我：PR 链接 + 验收报告 + **确定性生成、WA 不可影响的 diff 风险高亮**，等我点"合并"（W3 人确认）。

# 第二部分 三个核心子系统

## 2.1 知识蒸馏层 KDL

### 2.1.1 定位与边界

KDL 是数字分身的"长期记忆 + 可代答知识库"，只做三件事：**摄取 → 蒸馏 → 供检索**。它**不产生对外答复**（那是分诊 Worker 的事），只向 Worker 提供"带证据、带权威级、带新鲜度"的候选知识片段。

硬约束映射：自身无 LLM 写执行权（蒸馏中 LLM 只产"知识候选 JSON"，落库由无 LLM 的 Ingestor 按规则做）；代答用片段必可溯源；"升为权威知识"必须经本人确认这一确定性闸门；入库即脱敏 + 污点传播 + 出口管控。

### 2.1.2 四类知识源与统一知识单元 KU

| 源 | 代号 | 原始形态 | 蒸馏产物 | 新鲜度风险 |
|---|---|---|---|---|
| 在开发项目代码 | CODE | commit、函数/类/接口、README、配置 | 代码事实卡（symbol→行为/契约/坑） | **极高**（代码漂移） |
| 历史 issue / 写过的问题 | ISSUE | issue 正文、标题、标签、状态流转 | 症状→根因→处置 卡 | 中 |
| 历史问答对 | QA | 钉钉消息、文档评论、邮件往返 | 规范化 Q→A 对 | 中 |
| 干过的工作套路 | PLAYBOOK | 重复操作序列、审批/部署流程 | SOP/Runbook 步骤卡 | 低-中 |

**KU 数据模型（字段最小化）：**

```
KU {
  ku_id, source_type(CODE|ISSUE|QA|PLAYBOOK), title, body, body_redacted(bool),
  taint(CLEAN|INTERNAL|SENSITIVE),
  authority(DRAFT|REVIEWED|AUTHORITATIVE|DEPRECATED),
  public_ok(bool),          -- 是否预先核定"可对外公开"（吸收审查 A1）
  confidence(0..1),         -- 仅排序参考，不决定能否答
  freshness(FRESH|STALE|EXPIRED|UNKNOWN),
  provenance[],             -- 溯源，缺失则不可代答
  repo, commit_sha, file_path, symbol, line_range, content_hash,  -- 仅 CODE
  created_at, updated_at, last_verified_at, expires_at, superseded_by, owner
}

Provenance {
  kind(COMMIT|ISSUE_URL|MSG_ID|DOC_ID|MAIL_ID|FILE),
  ref, quote(脱敏后摘录), quote_taint, captured_at, retrievable(bool)
}
```

**规则：** `provenance` 为空 → `authority` 强制 DRAFT 且不得用于代答（对应"代答必可溯源"）。

### 2.1.3 摄取与蒸馏流水线

```
采集 Collector → 脱敏 Redactor → 蒸馏 Distiller(LLM,只产JSON)
   → 校验 Validator(无LLM) → 入库 Ingestor(无LLM,确定性, authority=DRAFT)
   → 索引 Indexer(FTS+符号+按需向量)
```

关键：**Distiller 是唯一用 LLM 的环节，输出严格 JSON、无任何副作用**；Validator/Ingestor/Indexer 全无 LLM、纯规则。

各源蒸馏要点：

- **CODE（抗漂移是核心）**：单元 = 符号级（函数/类/接口/导出 API），强制绑定 `commit_sha + file_path + symbol + line_range + content_hash`。蒸馏三类内容：①契约 ②行为/副作用 ③已知坑。**禁止凭空概括**：映射不到具体符号即丢弃。
- **ISSUE**：蒸馏为 `症状→根因→解决→关联代码符号`，与 CODE-KU 双向引用，代码改动可连带标 stale。
- **QA（自动配对，吸收审查 A5）**：配对器用规则 + 轻量 LLM 判定（不落库），信号 = 同线程疑问句/@我 + 我的后续回复、reply/quote、时间邻近、邮件 In-Reply-To。**强校验回复确实出自本人 account 且无中间插话**，防抢答污染。一律入库 DRAFT。
- **PLAYBOOK**：归纳触发条件→步骤→校验点→责任人。是干活 Agent 的意图模板来源，但本身 DRAFT，须本人确认才可被自动援引为 W1+ 依据。

### 2.1.4 代码漂移与新鲜度治理（重点）

| freshness | 含义 | 判定 | Worker 可用性 |
|---|---|---|---|
| FRESH | 绑定 commit 仍是最新、hash 匹配 | 校验通过 | 可作依据答（按权威级） |
| STALE | 符号在 HEAD 已变但仍存在 | hash 漂移 | 仅"参考性"答，必降级 + 提示"可能已过期，正在复核" |
| EXPIRED | 符号已删/文件不存/源不可取回 | 找不到符号 | **禁止代答**，转 abstain/升级 |
| UNKNOWN | 未绑定或从未校验 | 无 commit 绑定 | 视同 STALE 下限 |

**校验触发三层：** ①事件触发（监听 push/merge，对受影响 file_path 反查 KU 逐符号重算 hash）；②定时巡检（超阈值未验证批量复检）；③**检索时懒校验（最后防线）**：Worker 取用 CODE-KU 时实时轻量 hash 比对再返回。

> **吸收审查 D1（TTL/TOCTOU）：** 懒校验的秒级缓存对"登录接口 500"这类高频变更场景仍可能失效。修复：**高敏/高频域不用缓存**；对外答复附"基于 commit X"的免责，并在**发出瞬间做二次校验**，窗口内代码再变即拦截。

**漂移联动失效（污点式失效传播）：** 符号变更 → 直接绑定的 CODE-KU 置 STALE → 沿 ISSUE↔CODE、QA↔CODE 引用边把引用该符号的派生 KU 标 `derived_stale`。**AUTHORITATIVE 一旦底层代码漂移 → 自动降级为 REVIEWED + STALE** 并生成"待本人复核"任务。权威性不能跨过代码变更自动延续。

### 2.1.5 权威升级状态机（本人确认是唯一升 AUTHORITATIVE 路径）

```
DRAFT ──自动蒸馏──► (停在 DRAFT)
  │ 本人轻确认("对，可用")
  ▼
REVIEWED ──本人强确认("作为权威")──► AUTHORITATIVE
  │                                      │
  │ 代码漂移/本人否定                      │ 代码漂移
  ▼                                      ▼
DEPRECATED ◄── 本人否定 ──(任意态)   REVIEWED+STALE(回炉复核)
```

| authority | 谁可设置 | 代答可信度 | 对应代答级 |
|---|---|---|---|
| DRAFT | 机器自动 | 仅上下文参考 | 倾向 abstain/升级 |
| REVIEWED | 本人轻确认 | 可代答但标"待复核" | C1 |
| AUTHORITATIVE | **仅本人强确认** | 可较直接代答（仍标助理身份） | C0/C1 |
| DEPRECATED | 本人/系统 | 不可用 | 不可用 |

升级动作由本人在钉钉对"复核卡"点确认完成，事件写审计；**Executor 按规则改 authority 字段，无 LLM 参与**。

### 2.1.6 可溯源、abstain 与检索

**可溯源（硬约束）：** 每条 KU ≥1 条 provenance，否则锁死 DRAFT 且 `serve_blocked`。断链检测：定期校验 `retrievable`，来源被删/不可达 → 置 EXPIRED/`evidence_broken`，禁止代答。**吸收审查 A7：** `quote` 也必须过脱敏与污点档；对外只给"出处标识"，不直接展示原文 quote。

**Abstain（不确定就升级，绝不编）：** Serve 层返回结构化裁决 `Verdict{decision, kus, reason}`。强制 ABSTAIN 条件（任一命中）：
1. 无 KU 命中 / 命中全为 DRAFT；
2. 命中全 EXPIRED，或最佳为 STALE 且无 FRESH 替代；
3. 顶级候选 provenance 断链；
4. 候选互相矛盾（**仅 REVIEWED+ 才算冲突源，DRAFT 不参与对立面——吸收审查 A5 防升级 DoS**）；
5. 涉及承诺/决策/对外口径；
6. 检索 top1 相关度双低（语义 + 词面）。

> **吸收审查 A4（grounding gap）：** 检索回查只保证"引用存在"，不保证"答案=引用"。新增**无 LLM 的"答案-证据蕴含校验"**（NLI/字符串覆盖率门）：覆盖率不足强制 abstain；abstain 判定与拟答**用不同模型/不同 prompt**，降低共模注入。

**检索（FTS + 符号索引，按需向量）：**

| 层 | 技术 | 用途 | 启用 |
|---|---|---|---|
| L1 符号索引 | 倒排 symbol/file/repo→ku_id | 代码精确问题 | 始终（CODE 优先） |
| L2 FTS | SQLite FTS5/关键词 | 词面匹配 | 始终 |
| L3 向量 | 句向量 + ANN | 语义近似 | **按需**：L1/L2 不足时 |

融合排序 `score = w_f·freshness + w_a·authority + w_r·相似度`；EXPIRED/DEPRECATED 直接出局，STALE 强降权。freshness/authority 是硬门槛，相似度只在通过门槛者间排序。**吸收审查 C3：SENSITIVE 内容不进入面向外部提问的同一向量空间（分库），避免被语义检索召回到对外代答上下文。**

## 2.2 分诊与路由

### 2.2.1 整体数据流

```
钉钉入站(dws收消息, R0)
  │
[1] 归一化 & 去敏(Executor,无LLM)  统一InboundEvent，污点标注、@解析、引用链拼接
  │
[2] Case 归并器(Executor,无LLM,规则)  按(会话+人+主题指纹+时间窗)落Case
  │
[3] 分诊 Worker(LLM,只思考)  产 TriageIntent JSON
  │
[4] 路由 Executor(无LLM,按规则判级)
  ├─ C0 → 代答生成 → 出口管控 → 自动发(R≤2,过硬闸)
  ├─ C1 → 拟答 → 草稿待我确认
  ├─ 派活 → 干活Agent(W0-W2)
  ├─ C2/决策 → 升级我(DING)
  └─ C-abstain → 升级我 + 标注缺口
  │
[5] SLA 守护(Executor + Cron,无LLM)  超时→提醒/升级/再分诊
```

不变量：**[1][2][4][5] 全无 LLM；[3] 及下游生成只产意图/草稿**。LLM 输出永远先过 [4] 规则闸门再决定能否触达"自动发送"。

### 2.2.2 意图识别（多标签）

| 意图码 | 名称 | 判定要点 | 默认路由 |
|---|---|---|---|
| I-ASK | 提问/求知 | 怎么做/是什么/为什么 | 代答(C0/C1) |
| I-BUG | 反馈 Bug/故障 | 报错、不可用、"挂了" | 派活(W) + 代答确认 |
| I-TASK | 派活/请求执行 | "帮我改/加个/部署" | 派活(W0-W2) |
| I-DECIDE | 要承诺/决策 | 排期、资源、拍板 | 永久升级我(C2) |
| I-STATUS | 进度催问 | "好了吗/到哪了" | 查 Case 状态自动答(C0) |
| I-FYI | 知会/通知 | 仅告知 | 记录归档 |
| I-CHAT | 闲聊/情绪 | 寒暄、情绪 | 轻量代答或不响应 |
| I-AMBIG | 意图不明 | 信息不足 | abstain→反问/升级 |

附带产出：`need_my_response`、`target_is_me`（用通讯录 + 项目归属判断，非自封）、`evidence_refs`（可溯源）。

### 2.2.3 代答置信度分级 C（出口闸门）

| C级 | 含义 | 触发（全部满足） | 动作 | 自动发？ |
|---|---|---|---|---|
| **C0** | 事实型高置信 | 单一权威可溯源；无承诺无主观；脱敏后无污点；R≤2；**KU.public_ok=true** | 自动生成 + 发送，署"助理代答(可复核)" | 是（过硬闸） |
| **C1** | 需判断/拟答待确认 | 需综合多源或含倾向，但不构成对外承诺 | 草稿 → DING 我一键"发/改/撤" | 否 |
| **C2** | 承诺/决策型 | 涉排期、资源、对外承诺、拍板、跨团队、不可逆 | **永久升级我**，仅给建议草稿 + 利弊 | 永不 |
| **C-abstain** | 弃答 | 证据不足/冲突/无法溯源/高污点 | 反问澄清或升级并标缺口 | 否 |

**降级优先原则**：任一不满足上级条件即落更保守级；置信打分只辅助，**最终级别由规则闸门裁定**（不信自评）。

**C0 自动发送额外硬闸（Executor 校验，任一不过即降 C1）：**
1. `evidence_refs` 非空且可解析回真实出处；
2. 答案文本通过**承诺语义分类器**（见下，非纯关键词）；
3. 污点 ≤ 允许出口阈值；
4. 接收对象/群在允许出口白名单内（见 [3.2](#32-对外答问出口管控)）；
5. 同 Case 未处于"已升级我决策"挂起态；
6. 命中的 KU `public_ok=true` 且 freshness 通过发出瞬间二次校验。

> **吸收审查 A6（承诺检测）：** 承诺是语义不是关键词。"上线/保证/一定/x 号前"这类黑名单极易被"应该问题不大""这周内八成能好""放心"绕过。改为**语义分类器 + 保守默认**：任何含时间/数量/资源/对外义务的表述一律降 C1。宁可错杀。
>
> **吸收审查 A1（注入借力代答）：** C0 自动发的 KU 必须预先标 `public_ok`（可对外公开白名单），而非"未命中红线即可发"。**新问题即使命中老 KU，也不自动继承老 KU 的公开档位**；对入站文本做注入检测。

### 2.2.4 紧急度 U（驱动 SLA，吸收审查 C6）

| 紧急度 | 信号 | 首响 SLA | 升级方式 |
|---|---|---|---|
| U3 致命 | 生产故障/资损/全员不可用/客户投诉升级/上级@ | ≤5 分钟 | 立即电话级提醒 + 并行派活 |
| U2 高 | 阻塞他人、deadline 临近、关键人催 | ≤30 分钟 | DING 强提醒 |
| U1 中 | 一般问题 | ≤4 小时（工作时段） | 普通通知 |
| U0 低 | FYI/闲聊 | ≤1 工作日 | 静默入队 |

> **吸收审查 C6：U3 不可由攻击者纯文本自我声明。** 纯文本"生产挂了！！！"最多判 U2；**U3 须有客观佐证**（监控告警、可信来源）；U3 并行派活前置一个轻量真实性校验，避免电话轰炸 + 误派活。

### 2.2.5 Case 归并与状态机

归并键 `mergeKey = f(conversationId, rootSenderId, topicFingerprint, timeWindow)`：topicFingerprint 基于引用链/@关系/关键词与实体；默认 30 分钟静默窗内同主题归并；显式"回复引用"强制归并；发起人不同 + 指纹距离大则不并。

```

   入站 ──────────▶ │  待分诊 TRIAGE │
                    └──────┬───────┘
        ┌──────────────────┼─────────────────────┐
        ▼                  ▼                      ▼
 ┌────────────┐   ┌──────────────┐        ┌──────────────┐
 │ 待答 ANSWER │   │ 进行中 WORKING│        │待我决策 ESCAL.│
 │(C0自动/C1草稿)│  │ (派活W0-W2)  │        │ (C2/abstain) │
 └─────┬──────┘   └──────┬───────┘        └──────┬───────┘
       │ 已答/已确认       │ 验收通过/PR提交        │ 我已决策
       └──────────────────┴────────────────────────┘
                          ▼
                   ┌──────────────┐   对方追加新诉求/复发
                   │  已闭环 CLOSED │ ──────────────► 回 TRIAGE
                   └──────────────┘
```

附加状态位：`paused_waiting_human`（任何环节需人确认时置位，阻断该 Case 自动发送）、`SLA_breach`。

### 2.2.6 SLA 守护（Executor + Cron，无 LLM）

| 场景 | 触发 | 动作 |
|---|---|---|
| 首响超时 | 入 TRIAGE 超 U 级首响 SLA 未路由 | 提醒我；U3 电话级 |
| C1 草稿待确认超时 | 超阈值（U2:30m/U1:2h） | 二次提醒；超长且 U≤1→记"已读未答"留痕，**不擅自发** |
| WORKING 停滞 | 干活 Agent 超期未交付 | 提醒我 + 催办；连续 2 次→升级 ESCALATED |
| ESCALATED 等我超时 | 我未决策超阈值 | 阶梯提醒（钉钉→DING→电话级），**绝不替我决策** |

> **关键不变量：任何超时都不会把 C2/abstain 自动转成自动发送，超时只升级催促，绝不放宽出口闸门。** C1 在我确认前对方**零可见**（吸收审查 A3）。

## 2.3 双 Agent 编排（指挥干活 Agent）

### 2.3.1 核心立场：干活 Agent 是不可信外包

干活 Agent（下称 WA）是项目外、能写代码/动文件/能提交的强力执行体。根本假设：**WA 可能能力很强，但永远不可信**——可能误解需求、越权、幻觉式"已完成"、自评注水、被污染上下文劫持。三道闸门夹住它：

```
我(本人) ──升级决策──┐
分诊Worker ──产意图──> 确定性Executor(无LLM,判级,持token)
                     │  TaskSpec(脱敏/边界)        │ W0-W3 落地
                     ▼                            ▼
              ┌─────────────┐          ┌──────────────────────┐
              │ Worker-Agent│ 产diff/  │ 独立用户+容器沙箱      │
              │ (不可信外包) │ PR/声明  │ +egress审计+token隔离 │
              └─────────────┘          └──────────────────────┘
                     │ 自评(零信任)
                     ▼
        独立验收器(干净沙箱重跑测试/CI/diff静态核验,不读WA自评)
                     │ 真实结果
                     ▼
        闭环回报 → 助理署名R3对外发送 / 不过→转人工
```

- **入口闸 TaskSpec**：只给脱敏、有边界的任务，不给原始凭证/客户数据。
- **执行闸 W0–W3 + 确定性 Executor**：才能落地写操作。
- **出口闸 独立验收 + 出口管控**：WA 对外发声权为零。

### 2.3.2 移交协议 TaskSpec（入口闸）

| 字段 | 必填 | 含义 | 不可信防护点 |
|---|---|---|---|
| `task_id` | ✓ | 全局唯一，贯穿验收/回报/审计 | 防混淆/重放，产物按 task_id 绑定 |
| `goal` | ✓ | 一句话目标 | 只描述目标，不含商业背景 |
| `context` | ✓ | **已脱敏**的必要上下文 | 客户名→占位、密钥剔除、内部 URL→代号；污点字段不得进入 |
| `acceptance` | ✓ | 机器可判的验收标准 | WA 看得到标准，**判定权不在 WA**；**标准本身须确定性派生或本人核定（防注入，见下）** |
| `risk_bound` | ✓ | 本任务最高动作级 W0/W1/W2（W3 永不签发给 WA） | Executor 对照拒绝越级 |
| `scope` | ✓ | 允许触达的路径/仓库/分支白名单 | 越界写=硬拒绝 |
| `deadline` | ✓ | 截止时间；超时自动转人工 | 防无限挂起占资源 |
| `forbidden` | ✓ | 黑名单：禁改文件、禁调外网域、禁碰命令 | **默认含 CI/workflow/依赖锁文件**（见下） |
| `secrets_ref` | ○ | 仅必要时给"token 句柄"而非明文 | 凭证永不进 WA 上下文，且**句柄绑定动作模板 + 配额** |
| `report_to` | ✓ | 回报回收点（只回 Executor/验收器，非对外群） | WA 无对外发送目标 |

脱敏管线（context 生成强制）：分级脱敏（R0 原文 / R1 代号 / R2 占位符摘要 / R3 禁出→改升级）；污点传播继承最高级、不可洗白；签发前 Executor 正则 + 实体扫描，命中密钥/邮箱/客户实体即阻断签发并升级。

### 2.3.3 干活动作风险分级 W0–W3（执行闸）

| 级别 | 名称 | 典型动作 | 落地者 | 需人确认 | 隔离手段 |
|---|---|---|---|---|---|
| **W0** | 只读探查 | 读代码/测试/日志、git log | Executor 只读 token | 否 | 只读挂载，沙箱无写权 |
| **W1** | 本地改+测试 | 改代码、跑单测、构建、lint | Executor 写入 worktree 沙箱 | 否（产物隔离） | **独立用户 + 容器 + worktree + egress 全代理审计** |
| **W2** | 提 PR/草稿 | push feature 分支、开 draft PR、提文档草稿 | Executor 受限 push token | 否（**自动上限**） | PR 必 draft；不能 target 受保护分支 |
| **W3** | 合并/部署/发布 | merge、deploy、release、改生产/权限、对外正式发文 | **仅本人**经 R2/R3 确认 | **是（永久，绝无例外）** | WA 与 Executor 均无 W3 token；人在环 |

Executor 裁决（无 LLM，纯规则）：`动作级 > risk_bound` 硬拒；触达 scope 外或命中 forbidden 硬拒；W1 全程在沙箱、主工作区零污染；W2 强制 `draft=true` 且 base ∈ 非受保护分支；任何"想跨 W3"的意图一律转升级。

> **吸收审查 B1（PR 本身即攻击载体）：** draft PR 可在 CI/Actions 植入恶意 workflow（`pull_request_target` 类漏洞窃取 secrets）、改依赖清单引供应链执行、用描述社工诱导合并。修复：**forbidden 默认含 CI/workflow/依赖锁文件**，改动这些强制人工；**PR target 必须是无 CI 自动触发的隔离分支**。
>
> **吸收审查 B3（沙箱逃逸/外带）：** 沙箱必须是**独立用户 + 容器**，与主机不共享 `~/.dws`（不同 uid + 文件权限 + 不挂载）；**egress 完全代理审计**——连包镜像也走审计代理记录全部出站（防 DNS 隧道/把数据编码进包请求外带）。
>
> **吸收审查 B6（句柄间接滥用）：** `secrets_ref` 句柄须**绑定到具体动作模板与配额**，Executor 对句柄用途做规则白名单，而非"WA 说用就用"，防 WA 借句柄在 scope 内做有害事（刷垃圾 push、耗 API 额度）。

### 2.3.4 独立验收（不信自评）

> **铁律：验收器从不读取 WA 的"我已完成/测试通过"自评文本，只看可独立复现的客观证据。** 验收器是无 LLM 的确定性流水线。

| 维度 | 方法 | 通过条件 | 反作弊 |
|---|---|---|---|
| 功能 | 在**干净、无 secrets、无网、用后即焚**沙箱重跑 `acceptance.tests` | 全绿、指定 `-count` 稳定 | 重新 checkout WA 分支自己跑，忽略 WA 日志 |
| 回归 | 跑全量既有测试 | 无新增失败 | 对比 base 基线 |
| CI | 读 CI 真值（如 required） | CI 绿 | 读 CI 系统真值，非 WA 转述 |
| Diff 静态核验 | 解析 PR diff | 满足 diff_constraints（文件数/路径/无禁改/无新增密钥） | 静态扫描密钥与 forbidden |
| 边界合规 | 比对 scope/forbidden | 零越界 | 命中即否决 |
| 无副作用 | 沙箱网络/文件审计日志 | 无未授权外联/禁改 | eBPF/审计日志为准 |

判定与转人工：全 PASS→闭环回报；不过→自动重试 ≤N 次（失败证据**经脱敏后**塞回 context，仅"断言摘要"非原始日志——吸收审查 B4）；越界/命中 forbidden/检出密钥/试图 W3/超 deadline → 直接转人工不重试。**重试不得放宽 acceptance（放宽属决策，只能本人做）。** "自评通过但独立验收不过"记为 WA 可信度负分，后续同类任务自动收紧 risk_bound。

> **吸收审查 B2（验收标准被注入污染）：** acceptance 写在 TaskSpec、由分诊 Worker（LLM）生成，攻击者可经 bug 反馈注入，诱导生成"跑一个必过空测试"的验收门。修复：**acceptance 必须由确定性规则从需求派生或由本人核定，LLM 不得自由撰写验收门**；高风险任务的 acceptance 须人确认。
>
> **吸收审查 B5（人被诱导确认 W3）：** 带外签名只保证"是我点的"，不保证"我看懂了"。攻击者用伪造 U3 制造时间压力 + 看似无害的 PR + WA 影响的验收摘要话术。修复：确认卡必须展示**确定性生成、WA 不可影响的 diff 摘要与风险高亮**（改了哪些敏感文件、是否动 CI/依赖/权限）；高风险 PR 引入冷静期或第二人复核。

### 2.3.5 闭环回报（出口闸）

WA 对外发送权 = 0。所有对外消息由**助理身份署名代发**，经 R3 出口管控；内容**必须基于真实验收结果**，不引用 WA 自评话术；携带 task_id + 验收证据指纹（测试报告 hash、CI run id、PR 链接、diff 摘要）。

| 场景 | 出口动作 | 分级 | 署名/确认 |
|---|---|---|---|
| 验收通过，告知发起同事 | 钉钉回复 | R3 | "助理代发·待本人复核"：附 PR + 验收摘要 |
| 验收通过，需推进 W3 | 提请本人合并/部署 | 升级 | 决策卡：diff + 验收证据 + 一键确认 |
| 验收不过/越界/超时 | 转人工通知 | R3 + 升级 | 助理署名："未通过自动验收，已升级，暂不处理" |
| 内部状态流转 | 更新待办 | R1/R2 | 内部，免对外署名 |

**对外回报模板：**

```
助理代答(待本人复核)
任务 WA-2026-0618-014：svc-order 重复扣减库存
状态：已通过独立验收(非Agent自评)
证据：并发用例×20 全绿 | CI #4821 绿 | 改动 3 文件,限 src/order/**
产物：Draft PR #312(尚未合并,需本人确认后方可合入)
下一步：已请本人复核合并(W3),合并/上线由本人确认。
```

### 2.3.6 不可信攻防小结

| WA 可能的坏行为 | 防线 |
|---|---|
| 谎称"测试已过" | 验收器无视自评，干净沙箱重跑 |
| 偷改禁改文件/迁移脚本/CI/依赖 | forbidden 黑名单（含 CI/依赖锁）+ diff 静态核验 + 边界审计硬拒 |
| 越权直接合并/部署 | 无 W3 token，Executor 拒绝，人在环 |
| 上下文注入泄露客户数据 | context 已脱敏，WA 拿不到原始敏感数据 |
| 偷偷外联回传 | 沙箱独立用户 + 容器 + egress 全代理审计 |
| 无限挂起占资源 | deadline 超时自动转人工 |
| 反复刷重试绕过验收 | 重试上限 N 且不得放宽 acceptance |
| 直接对外发消息制造既成事实 | 对外发送权为 0，唯一出口经助理署名 R3 |
| 污染验收标准 | acceptance 须确定性派生/本人核定 |

# 第三部分 安全 / 隐私 / 认证 / 质量

## 3.1 PolicyGate：三套分级统一裁决

PolicyGate 是**唯一**把三套分级合并裁决并做对外出口脱敏的确定性组件。它不思考、不调 LLM，只对输入意图 JSON 做规则匹配、查表、字段扫描、签名校验。

```
分诊Worker / 干活Agent ──意图JSON──► ┌──────────────────────────┐
                                     │      PolicyGate(无LLM)     │ ← 唯一收口
                                     │  统一裁决 R×C×W + 出口管控  │
                                     └──────────────────────────┘
                  ┌──────────┬──────────────┬─────────────┐
                  ▼          ▼              ▼             ▼
              自动放行    暂存/草稿      升级给本人      拒绝/熔断
```

**三维取最严（短板原则）：** 每条意图同时打 `(R, C, W)` 标签，最终动作 = 三维各自允许动作的最小交集。

```
decision = MIN_BY_STRICTNESS( resolve_R(...), resolve_C(...), resolve_W(...) )
严格度序：AUTO < DRAFT < HUMAN_CONFIRM < DENY
```

**裁决矩阵（代表性组合）：**

| 场景 | R | C | W | 合并结果 |
|---|---|---|---|---|
| 群里问功能怎么用，答案来自公开文档 | R2 | C0 | - | DRAFT→过出口管控后自动发 |
| 同上但涉及未发布计划 | R2 | abstain | - | **DENY 代答正文** + 升级 |
| 反馈 bug，Agent 定位改好提 PR | R1 | C1 | W2 | W2 自动提 PR + C1 待复核回复 |
| 要求"直接合并上线" | R3 | - | W3 | **HUMAN_CONFIRM（带外）** |
| 问"导出包对不对"，纯读验收 | R0 | C0 | W0 | AUTO |
| 要改他人审批/撤回 | R3 | - | - | **HUMAN_CONFIRM（带外）** |

**额外门槛：出口管控优先级高于置信度。** 代答即使 C0，只要触达 R2（对外群发），必须先过出口管控（[3.2](#32-对外答问出口管控)）再放行；任一红线命中即降级，绝不因 C0 豁免。

**永久人确认终态（不可被任何自动路径覆盖）：** R3 全部、W3 全部、C 维度对外承诺性回复、出口管控命中"密钥/内部代码/未公开计划"、dws never 项。

> **吸收审查 C1（根问题）：** PolicyGate 只做**结构性判级**（动作类型、目标分支、文件路径、API 名）；**语义性判级**（承诺、敏感、grounding）放在思考侧、默认保守降级，PolicyGate 只消费已标注的保守标签做 AND。两类判级 AND 后取严，确保无 LLM 组件不独自承担语义判断。

## 3.2 对外答问出口管控

代答正文落地**前的最后一道确定性闸**，独立于置信度。三步：**身份定级 → 颗粒度裁剪 → 红线扫描**。

**身份/群信任档：**

| 信任档 | 典型来源 | 可答颗粒度上限 |
|---|---|---|
| T0 外部/未知/外部群 | 非本企业、客户群 | 仅公开口径 |
| T1 内部普通同事 | 本企业、非项目组 | + 一般进展、非敏感答疑 |
| T2 项目组/直接协作方 | 同项目成员 | + 项目内进展、接口约定（不含密钥/未公开计划） |
| T3 本人/受托代理 | 仅本人带外身份 | 全量（写操作仍受 R/W 约束） |

> **群 > 个人取更严，且以"信息会被谁看到"为准（吸收审查 A2）：** 外部群里即便对方是内部同事，按 T0 群上限裁剪。进一步修复身份假冒与转发外流：
> - **T2/T3 级答案禁止进入任何可转发/可被外部加入的群**；
> - **新成员入群触发历史敏感消息撤回评估**；
> - 高敏答案走单聊 + 水印 / 阅后即焚；
> - 身份定级以"信息可见面"而非"对谁说"为锚点（污点不随钉钉转发传播，故对外流出的内容必须本就可公开）。

**颗粒度裁剪：** 按信任档白名单放行字段，其余删除/泛化（如 T1 把"具体上线日期"泛化为"正在推进中"）。

**红线扫描（命中即剥离并降级，绝不外发）：**

| 红线类别 | 检测方式 |
|---|---|
| 密钥/凭证 | token/AKSK/私钥/连接串正则 + 熵值检测 + `~/.dws` 字段比对 |
| 内部代码 | 代码块/内部路径/仓库名/未公开类名/堆栈匹配 |
| 未公开计划 | 标注 internal/未发布的知识条目命中 |
| 污点传播 | 任何 tainted 来源内容，标签随文本传播，出口按最敏感档处理 |

每条对外答问留**可溯源审计**（来源 KU ID、信任档、裁剪规则、红线结果、nonce/签名），附"助理代答/待本人复核"标识。

## 3.3 Kill Switch 与带外确认通道

**Kill Switch（熔断）：** 两种触发，效果都是 PolicyGate 立即把非 R0 动作降为 DENY，只留只读与升级。

| 类型 | 触发 | 行为 |
|---|---|---|
| 前置硬确认 | 带外 CLI `dws-gate kill --scope=all\|answer\|work` | 写 `~/.dws/gate.lock`（Keychain 加密），每次裁决前先读锁 |
| 频次熔断（自动） | 单位时间动作超阈值（5min 内 R2+ > N、同群代答 > M、W2 提 PR > K） | 自动置锁 + 升级，需带外解锁 |

> **吸收审查 D2：** 熔断分级——区分**降级到只读** vs **全停**，区分"攻击触发"与"自身故障"；阈值**按域而非全局**，防攻击者主动触发熔断瘫痪分身（用熔断本身做 DoS）。

**带外确认通道（绝不解析群文本）：核心原则——确认信号永不来自钉钉消息体**，防社工/注入。

```
1. PolicyGate 产待确认项 → 写 pending 队列，生成 nonce(一次性、随机、绑意图哈希)
2. 经 DING/本人专属私聊 仅推送「确认链接指向本地CLI + nonce摘要」(通知,非授权入口)
3. 本人本机执行：dws-gate approve --nonce=<n> --sig
   本地CLI 用本机 Keychain 私钥对 nonce+intent_hash 签名
4. PolicyGate 校验签名 + nonce未过期未用过 + 意图哈希一致 → 解冻 → 交 Executor 执行
```

nonce 一次性、TTL 短（约 10 分钟）、与意图哈希强绑定，改一字节即失效。群里即使原样贴出 nonce 也无法伪造签名。kill/unlock/R3/W3/never 项**只能**走此带外签名通道。

## 3.4 认证与凭证隔离

| 机制 | 做法 |
|---|---|
| 单实例持有凭证 | 单实例持有 `~/.dws`，文件锁 + 刷新，防 token 撕裂 |
| never 名单 | `auth export/import/logout/reset` 判 R3 并直接拒绝，只走带外 CLI |
| 凭证与沙箱隔离 | Executor 持 `~/.dws`，WA 沙箱**不同 uid + 文件权限 + 不挂载**，沙箱内 `~/.dws` 不可达 |

> **吸收审查 C2（单点）：** 单实例崩溃=分身全停，对"及时响应"致命。补充：定义**锁超时/抢占/healthcheck/优雅降级到只读**；明确凭证持有进程与沙箱进程的 uid 边界。

## 3.5 落盘、脱敏、污点传播

| 护栏 | 做法 |
|---|---|
| 落盘加密 | 凭证与敏感数据进 Keychain 加密 |
| 字段最小化 | 只存判级与溯源必需字段；落盘字段 ⊆ 白名单 schema |
| 分级脱敏 | 密钥/token/内网 host/PII/客户名 → 占位符；可逆需求的密文存独立 Keychain 保险库，KU 只存占位 |
| 污点传播 | 派生数据取上游各源 taint 最高级，不可洗白 |
| 出口管控 | 所有对外回话经统一出口，加标识、查污点、查承诺 |

> **吸收审查 C3：污点是系统内概念，出系统（钉钉转发、PR、CI 日志、向量库）即不可控。** 故 **SENSITIVE 内容永不出系统**；SENSITIVE 不进入对外可检索的同一向量空间（分库）。

## 3.6 质量进化（只落数据不落代码）

进化只动数据（eval 金标准、检索语料、分级阈值走数据仓），改代码逻辑需评审解冻。

| 质量门 | 机制 | 落点 |
|---|---|---|
| 确定性回查 | 答案片段指纹比对原始源，失败即弃 | 全程 |
| 答案-证据蕴含校验 | 无 LLM 的 NLI/覆盖率门，不足强制 abstain | Serve/出口 |
| eval 金标准 | 来源 = 我对 draft/PR 的修改，自动入数据集 | 数据仓 |
| 指标独立性 | 评测器/验收器与 Worker 不同实例、不读自评 | 独立服务 |
| 高影响域全审 | 域路由表强制 C0 关闭、全人审 | 配置数据 |

评测指标（独立于自评）：采纳率、编辑距离中位数、abstain 准确率（漏报=幻觉风险）、溯源完整率。

> **吸收审查 C4（投毒/漂移）：** 闭环学习可被长期投毒（诱导我在特定话题反复以某方式回复，或用升级洪水逼我草率标注）。修复：**回流样本经独立审计 + 配额**；关键域金标变更**双人复核**；监控行为分布漂移；数据变更本身设评审门（数据即新攻击面）。

# 第四部分 架构与路线图

## 4.2 分阶段路线图

**总原则：** 后一阶段准入门槛 = 前一阶段退出条件全部实测通过 + 人工签署。退出条件必须是**可自动测量的数字**。贯穿不变量见 [附录 A](#a-决策基线不可推翻清单)。

| 阶段 | 目标 | 核心组件 | 退出条件（可实测，节选） |
|---|---|---|---|
| **0 地基+认证+安全门** | 凭证隔离、落盘加密、污点、出口骨架先立 | AuthBroker / Vault / TaintEngine / EgressGuard / RiskClassifier | 并发抢锁 100% 串行 0 撕裂；never 拒绝覆盖 100%；磁盘明文 grep=0；污点不丢=100%；绕过 EgressGuard 路径=0 |
| **1 知识蒸馏+检索（纯只读）** | 蒸馏可检索可溯源知识库，**绝不对外吐字** | KDL 全链 + 确定性回查 | 金标 Top-5 命中 ≥0.85 且均过回查；溯源抽样 100% 可点回；对外发送计数=0；脱敏抽检误/漏=0 |
| **2 分诊+拟答（我确认才发）** | Worker 分诊→拟答→进我确认队列 | 分诊路由 + C 分级 + eval 闭环 | C0 采纳率 ≥0.9；abstain 漏报率=0（无证据陷阱集）；无引用 draft 拦截 100%；C2/abstain 自动发=0 |
| **3 双 Agent 编排（W0/W1→W2）** | 指挥干活 Agent，先调研后提 PR | TaskSpec + W 分级 + 独立验收 | W0/W1 抽检达标；W2 验收通过率达标且自评-验收一致性监控；W3 自动合并=0；通知 100% 经出口管控 |
| **4 受控自治（逐域开放）** | 按域逐个开放 C0 自动代答 + 低风险派活；**绝无全局全自动开关** | 域路由表 + 持续 eval + 全局熔断 | 已开放域错答率 ≤阈值，一次"冒充承诺"即回滚到阶段 2；进化代码变更=0；任一指标越线自动降级该域 |

**逐域开放矩阵（阶段 4）：**

| 域风险 | C0 自动代答 | 低风险派活 W0/W1 | W2 | C2/W3/R3 |
|---|---|---|---|---|
| 低影响域 | 开放（达标后） | 开放 | 提 PR + 独立验收 | 永久人确认 |
| 中影响域 | 灰度（采样回查） | 开放 | 提 PR + 全验收 | 永久人确认 |
| 高影响域 | **永不自动，全审** | 仅 W0 | 仅人发起 | 永久人确认 |

**终态守恒（任一阶段成立）：** C2/W3/R3/auth 系列 = 永久人确认/never，无任何阶段可解除；对外文本唯一出口 = 出口管控，身份前缀不可绕过；任一 LLM 组件输出 = JSON 意图，执行权 100% 归无 LLM 的 Executor。

# 附录

## A. 决策基线（不可推翻清单）

| # | 基线 | 说明 |
|---|---|---|
| 1 | 思考与执行分离 | LLM 只产意图 JSON；执行归无 LLM 的 Executor；token 隔离靠 OS 权限非 PATH |
| 2 | 代答身份 | 一律"助理代答/待本人复核"，绝不冒充本人承诺；本人可一键撤回更正 |
| 3 | 干活上限 W2 | 提 PR/草稿封顶；W3（合并/部署/发布）永久人确认 |
| 4 | abstain + 可溯源 + 独立验收 | 不确定就升级绝不编；代答必溯源；干活必独立验收不信自评 |
| 5 | 认证隔离 | 单实例 + 共享 ~/.dws + 刷新文件锁；auth export/import/logout/reset 列 never |
| 6 | 三套分级有永久人确认终态 | R3、W3、C2/abstain；绝无全自动终态 |
| 7 | 数据治理 | Keychain 加密、字段最小化、分级脱敏、污点传播、出口管控 |
| 8 | 判级分两类 | 结构判级归 Executor，语义判级归思考侧保守标签，AND 取严 |
| 9 | 进化只落数据不落代码 | 语料/金标/阈值=数据；改逻辑需评审解冻 |
| 10 | SENSITIVE 永不出系统 | 污点出边界即失控，故最敏感内容禁止外发与对外可检索 |

## B. R0 只读白名单

> R0 = 纯读、无副作用、可全自动。仅以下读类调用免人确认；任何写/治理动作不在此列。

| 类别 | 示例 dws 读操作 |
|---|---|
| 消息 | 读会话消息、读群消息、读消息引用链 |
| 通讯录 | 查人员、查部门、查群成员（用于身份/信任档判定，非自封） |
| 文档/表格 | 读钉钉文档、读在线电子表格、读知识库条目 |
| 待办/日程 | 读待办列表、读日程（不创建不修改） |
| 审批/日志 | 读审批实例状态、读日报周报（只读） |
| 项目代码 | 代码仓只读拉取、git log/blame、读 CI 状态真值 |
| 云盘/听记 | 读云盘文件、读听记纪要 |

## C. never 名单

> 永不自动、连"人确认自动通道"都不走，只能本人在本机带外 CLI 显式操作。

| 项 | 原因 |
|---|---|
| `auth export` | 凭证外泄风险 |
| `auth import` | 凭证替换/注入风险 |
| `auth logout` | 使分身全停 |
| `auth reset` | 清空凭证/不可逆 |
