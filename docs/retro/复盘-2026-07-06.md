## 过程中暴露的问题

### Agent 侧（分身自己可以改进的）

### 1. 文档"三柱子结构"最初不对称——设计上讲三根柱子，写时只对 2 根有独立章节

项目说明.html 早期版本第 3、4 部分讲编排（6 道关卡 + 助理团队），第 5 部分讲进化，**知识层完全没有独立章节**——只在 1.1 里一句"图书馆"带过。这是"从助理身份出发写作 → 把编排细节详展开"的自然偏向，但设计层的三柱子对称被打破了。**用户直接指出后才补齐**——说明写文档时缺乏"对照设计骨架检查小节完整性"这一步。

### 2. 项目定位是渐进对齐的，不是一次定死——早期通俗版全部按"助理"框架写完，后来发现是"分身"

早期通俗版都是"雇一个 24 小时私人助理帮你干琐事"这个框架写的，跟"你继续做你的工作，AI 顺便帮忙"的常见 AI 助手定位一致。用户明确指出："**不是这样——分身干几乎所有事，我的角色变成 4 件事**"。这一改动导致：
- 顶部一句话点题重写
- 1.1 是什么完全重写
- 三个通俗版的类比（助理 / 秘书）都要考虑是否升级为"分身 / 员工"
- PPT 里的 slide 04 也是"24 小时在岗的私人助理"老框架

<em>如果一开始就锁定"分身"定位，很多措辞可以少改一遍。</em>但反过来说：定位是通过写作过程发现的——写到"1.1 是什么"时用户看到框架不对才提出，说明**"写出来"是澄清定位的必要动作**，不是浪费。

### 3. 钉钉引用散落 17 处——语言里的"隐性假设"

从"钉钉 = 项目主战场"演化到"钉钉 = 传声筒"再到"完全去钉钉化"这三轮过程中，早期语言里深埋的钉钉假设散落各处：
- `钉钉文档` × 3 处（1.2 / 1.3 / 3.5 / 关卡 4 表格）
- `钉钉群` × 1 处（3.3 事故背景）
- `发钉钉通知` × 4 处（1.2 表格 / 3.2 团队 / 3.4 表格 / 3.5 表格）
- `DwsReader 钉钉封装` × 1 处（3.5 表格）
- `阶段 2 · MVP 钉钉代答` × 1 处（第五部分）
- `钉钉里是不是真的收到消息` × 1 处（4.4 结尾）
- 一句话点题的 `它跟钉钉的关系` 整段
- 1.1 结尾"它跟钉钉的关系——历史包袱先说清"整段
- 1.2 钉钉侧痛点表整块
- 1.3 演化故事"最初钉钉分身"整章
- 收尾"给我回钉钉消息的分身"

每一处都要单独 sweep 一遍——这次共 15+ 次 Edit 操作才清理完。**说明重写章节标题和一句话点题不够——旧概念在语言颗粒里根深蒂固**，需要 grep 一遍确保清扫干净。

### 4. 子代理清单混淆了 4 种不同来源——读者容易误以为"分身团队 = .claude/agents/"

原 3.2 表格把 7 个东西并列：uat-deploy / uat-verify / prod-verify（**项目自定义 `.claude/agents/`**）+ Explore / Plan（**Claude Code 内置**）+ feedback 巡检 / ks_logs.py（**独立脚本 `scripts/`**）+ ClaudeCenter Worker（**远程外包服务**）。**这 4 种来源本质不同**，但表格里没区分。用户观察到"`.claude/agents/` 只有 3 个 md 但文档说 7 个"——问出来才发现要分类。

<em>写清单时的默认假设："都是 subagent 就都放一起"是错的——它们的物理位置、生命周期、维护方式都不同。</em>

### 5. 4.4 举例跟主题脱节——"分身自己变聪明"的例子不能是"脚本 bug 修好了"

进化层的 4.4 原本用 feedback_patrol 静默失败 → 修 script 的例子。**这是"编排层的系统级教训"（怎么避免静默失败），不是"进化层的分身能力提升"（分身踩坑后下次自动避开）**。用户识别出来："这个例子好像和复盘后改进分身能力没有关系"——完全对。换成"验收看最终产物"这条便签的来历后，才真正展示了"踩坑 → 复盘 → 便签 → 分身行为变化"的完整闭环。

**分辨标准**：讲进化层的例子必须能回答"分身之前会做 X（错），现在会做 Y（对）"这个问题。修 script 不满足这个标准（修的是脚本，不是分身）。

### 6. Maven 冷启动 26 分钟——build 时长预期外

后端 uat-deploy 跑了 26 分钟才完成本地 build（Maven 冷下载依赖 + classfinal 加密）。前端只需 70 秒。中途因为背景 agent 报"still downloading"多轮，我一度以为出问题了，实际是正常冷启动等待。**如果 uat-deploy agent 一开始能报出"Maven 冷启动预计 15-30 分钟"的时长预估**，我可以在等待时并行做别的事，而不是频繁 poll。

### 7. E2E FAIL 的诊断链路要多层展开——不能只报"FAIL 了"就完事

uat-verify agent 报 PARTIAL：前端 3 case 全绿、后端 FAIL。**如果就此报告给用户"后端有问题"，会误导用户以为 PR #128 有 bug**。实际去查 uat2 dataset pod 日志才发现是 uat2 环境的 OSS access key 失效——跟 PR 完全无关。**agent 报 FAIL 时，需要追一层根因才能给准确结论**，特别是环境类问题（OSS/数据库/依赖服务）常伪装成代码问题。

### 协作侧（人 + 分身一起可以改进的）

> 【新增维度】复盘不只是 agent 从错误里学，也是**双方从协作里发现"下次两边都可以怎么做得更顺"的模式**。
> 这不是 agent 评判用户——是**分身作为镜子帮用户看到自己没意识到的模式**。就像现实里两个搭档合作久了发现"你俩配合有个地方可以调"，不是教育，是镜子。语气原则：所有观察都要 framed 为"我们下次可以怎么约定"，不是"你有个问题"。

### 8.1 用户"OK / 按此方案"里可能藏着多个未澄清的决策点

**观察：** 定位调整那次，用户说"按此方案 + 英文目录 + 保留文件名 + 全部同 commit" —— agent 以为是"完整同意，全走"，但过一会儿发现英文 vs 中文目录用户其实希望进一步讨论。类似模式在 3 处出现（PPT 大纲、目录重组、KDL 合并方案）。

**共同点：** 用户回复简短（"OK / 按此 / 继续"），如果之前上下文有 3+ 个决策点，agent 假设"全部同意" → 有一部分不对 → 后来要改。

**协作规则候选：** 用户简短回复时，如果上下文有 3+ 个决策点，**agent 主动列一遍**"我理解你同意的是 1/2/3——哪个要改说一声"。这一步的成本 = 5 秒，收益 = 避免后续多轮修改。

### 8.2 环境类任务用户容易忘记附带健康状态

**观察：** 派 uat-verify 之前用户未告知 uat2 OSS access key 失效。E2E 报 FAIL 后 agent 追一层才发现是环境问题（PR 代码没事）。跨会话累积：07-06 KS_TOKEN 过期、07-06 uat2 OSS 失效、06-30 uat1 dataset-sso 未部署——同类模式已 3 次。

**共同点：** 用户假设"环境常规是 OK 的"，因此不主动告诉 agent；但环境凭证 / 部署状态实际会漂移。

**协作规则候选：** 涉及 uat / 生产 / 环境相关任务，**agent 在开工第一步做 quick health check**（OSS access key / DB 可连 / 关键凭证是否过期），报当前环境状态再动手。用户不再需要每次记着附带"环境 OK"——agent 自己查。查出问题就主动提示"要不要先修环境再干这个活"。

### 8.3 用户对 subagent 报告的信任度不一致

**观察：** uat-verify 报"PARTIAL"时，用户问 agent "E2E 测试完了吗" —— 把子代理报告直接当结论。但另一些场景 uat-verify 报 PASS 时，用户要求补截图证据。**信任度取决于结论方向**（PASS 要严 / FAIL 直接信）——但 FAIL 本身也可能是环境问题，直接信会误导决策方向。

**共同点：** 用户把子代理 FAIL 报告当"故障"信号，但没检查"故障原因是不是它自己该负责的范围"。

**协作规则候选：** 子代理报 FAIL 时，**agent 主动追一层根因**（是代码 / 环境 / 依赖）再给用户报告"FAIL + 根因分类"。用户接到 agent 二次转报时，可以直接看到"这是不是我该改代码的信号"。

### 8.4 用户偶尔一次派多个未排优先级的活

**观察：** "帮我把这两个 task 建 draft + 打两个 uat2 镜像 + 我推上去后跑 E2E" ——三件事，agent 要猜前后依赖 + 检查点。整个 session 里类似的 "batch 派活" 出现过 4-5 次。

**共同点：** 用户脑子里有清晰的执行顺序，但没显式说；agent 猜对是运气，猜错要返工。

**协作规则候选：** 用户一次派 3+ 步的活时，**agent 主动列执行计划 + 检查点**（"我打算 1 → 2 → 3，中间在 X 处停等你确认"），用户可以调整或直接放行。这一步 = 一句话，避免"步骤 2 走完才发现你希望先跑步骤 3"。

## 可以改进的措施

### Agent 侧（分身能改的：加便签 / 改 prompt / 加代码）

### 1. 建立"文档结构对称检查"清单

设计里的"三柱子"、"四角色"、"五阶段"这类结构，写文档时必须**先列出对应的章节骨架**再往里填内容——避免"写着写着某根柱子被跳过"。放进 CLAUDE.md 或写作类 skill 的提示词里，作为文档 review 阶段的强制检查。

### 2. 项目定位变更时的"联动更新清单"

每次项目定位变（"钉钉分身 → 平台 → 分身"这种量级的），伴随的联动更新清单固定成模板：
- 顶层：CLAUDE.md / README.md / MEMORY.md 索引 / 主页 index.html
- 通俗版：项目说明 + 3 根柱子通俗版 的一句话点题 + 核心类比
- 技术版：设计方案 1.1-1.3 + 分阶段路线图 + 各柱子技术文档的定位段
- PPT / 分享材料
- Memory：把"当前定位"写进一条 project 类 memory

这样定位改动能一次到位，避免半年后发现文档四处不一致。

### 3. 每次重要概念删除后跑一次 grep 确认清理干净

删掉 Part 2 三条底线之后我又发现好几处"§2.2 分家"的内部锚点链接需要更新到 §3.1。类似的，去钉钉化后跑 `grep -c '钉钉'` 确认 = 0 才算完。这个动作作为**结构性重构后的固定收尾步骤**。

### 4. 提炼一条 memory：agent 报 FAIL 时先追一层根因

新建 memory 条目：**"子代理报 FAIL 时不要直接转达 FAIL——先追一层：是代码问题、环境问题、还是外部依赖问题？"** —— 具体触发场景：uat-verify / prod-verify / 任何验收类 subagent 的 FAIL 结论。**Why：** 2026-07-06 uat-verify 报后端 FAIL 但根因是 uat2 OSS access key 失效，跟 PR 无关。如果直接转达会误导用户去改代码。

### 5. 提炼一条 memory：分身团队清单要按来源分类

在 CLAUDE.md 或 docs 里加一个"分身团队 = 4 类来源"的固定表述——项目自定义子代理 / Claude Code 内置子代理 / 独立脚本 / 远程外包。避免下次写文档又把它们混在一起。同步：在 `.claude/agents/README.md` 里加一句"这个目录只包含项目自定义 subagent，通用工种走 Claude Code 内置，不在这里出现"。

### 6. 派 subagent 时要求它报预期时长

新建规则：派 uat-deploy / build 类活儿时，subagent 一开始要**报出预计时长**（尤其是"冷启动 vs 缓存命中"这种量级差异）。这样调用方（主 Claude 或用户）可以并行做其他事，不用瞎等或反复 poll。可以作为 subagent 提示词的补充。

### 7. HiQLCD 1.2.0 空导出案例要沉淀

这一起 bug 是"3 层套娃 + 全流程静默"的典型——UI 允许选不存在版本、calculate 返回垃圾 H5、dataset 服务不失败——每一层都遵守了"看似正常"，全流程走完却是空产物。这跟"验收看最终产物"便签是同一模式的多次重犯。可以：
- 写进复盘留档
- 提取为一条新 memory：**"链路上任何一层'没数据但不失败'的行为都是隐藏 bug 温床"**（现有 `silent-catch-hides-bug` 便签的补充版本，专门盯"空数据被当成成功"这个模式）

### 协作侧（人 + 分身一起约定的：升级 memory 契约 + 落 collaboration 便签）

### 8. 升级 memory 契约：新增第 5 类 `collaboration`

在项目 CLAUDE.md 里补一个"双向复盘 · 协作类 memory"章节，说明本项目在系统自带的 4 类（user / feedback / project / reference）之外，额外使用 `collaboration` 类型——**记录"人 + 分身协作模式的观察 + 双方约定的协作规则"**。

**语气原则明写进 CLAUDE.md：** 协作规则不是 agent 评判 user——是"我们下次可以怎么约定"。**每条 collaboration 规则用户 review 后才生效**（同 human-in-loop 门槛）。

**触发时机明写进 CLAUDE.md：** 复盘时按"Agent 侧 + 协作侧"两个维度分别写；协作侧的观察由 agent 提出（跨会话累积 N 次同类模式），user review 后决定是否落 collaboration 便签。

### 9. 首批 4 条 collaboration 便签

对应 8.1-8.4 的观察，落 4 条 collaboration 类型 memory：

### 10. 未来定期的"协作模式复盘"

除了事故触发的复盘，未来增加一个**每 15-20 次 session 一次的"协作模式复盘"**——不是复盘某次事故，是回顾"这段时间我们协作里有什么可以更顺的地方"。由 agent 主动发起（观察到跨会话累积 3+ 次同类模式时提），用户 review 后决定要不要固化成 collaboration 规则。

**opt-in 机制：** CLAUDE.md 里加一个开关 `collaboration_review_enabled: true`（默认开）。如果用户不想被观察，改为 false。这是尊重协作模式本身也是协作偏好的一部分。
