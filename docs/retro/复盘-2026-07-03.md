## 做了什么

### 1. 四仓库一次性发布：accessScope 数组化 + 支付扩展 + 订单上链重试

把分散在多个长期分支（`feature/uat1-0530`、`feature/uat1-fix-blockchain-order-0520` 等）、迟迟没合到 `main` 的三条功能线（accessScope 权限数组化改造、Alipay/WeChat Pay 支付扩展、订单上链 3 次重试）抽取整合，分别提交到 dataset-sso / hiq-backend-admin / HiQ-backend-web / square-web-next 四个仓库的 `main` 分支，走 uat1 验证 → 生产发布的完整流程。

### 2. 处理三起生产事故（同一根因家族：Jackson 反序列化不容忍格式演进）

- **licenseContent 解析崩溃**：`ts_license` 表被两条证书生成流程共用、JSON 结构不同，导致 `OrderLicenseServiceImpl` 用错误的模型去解析 → PR #57 加类型判定跳过。
- **DataScope 解析崩溃**：accessScope 数组化回填后，`scope` 字段从标量变数组，老的 `DataScope.java`（无 `@JsonIgnoreProperties`、字段类型是 `String` 而非数组）直接抛异常 → 兼容新旧两种格式 + 实施 fail-closed 权限切换（PR #48/#49）。
- **`Ecoinvent_v 3.10.0` 类型不匹配崩溃**：一条"甄别·校正"回填脚本把 `version` 从标量改成数组（语义正确的修正），但生产代码还没跟上新格式，导致同样的崩溃——**数据先于代码升级**，是与前两起相反方向的同一类问题。

### 3. 抽取"失联"功能：用户行为看板

排查发现 PR #44/#53（后端）、#7/#14（前端）合并到的是长期分支而非 `main`，抽取重提为 #58/#16，过程中额外发现并修正两个真实 bug（迁移文件版本号冲突、产品编码写错 `hiq_backend`→`backend_admin`）。顺带排查同一分支上还有 API Key 管理、数据包资产管理两个功能也未上 main，按用户决定本次不处理，记录在案。

### 4. 发现并修复 feedback 巡检"看似正常、实际全部静默失效"

用户观察到"24 小时内 feedback 仓库出了很多新 issue，但一条钉钉通知都没收到"，追查发现是两层叠加的静默失败：launchd 环境下 `python3` 解析到没装 PyYAML 的 Homebrew 版本导致 `dws send` 直接崩溃，而 `_notify()` 用 `check=False` 从不检查发送结果，崩溃被完全吞掉、issue 照样被标记为"已见"。36 个历史"已见" issue 大概率一条都没真正送达。

## 过程中暴露的问题

### 1. 长期分支是"功能坟场"——合并到长期分支 ≠ 合并到 main

这次几乎所有要发布的功能（accessScope、支付、上链重试、行为看板、API Key 管理、数据包资产管理）都曾经过 PR review 并"合并"，但合并目标是像 `feature/uat1-0530` 这样的长期集成分支，而不是 `main`。这些分支本身又互相纠缠、多次 rebase 产生大量重复 commit（今天处理 PR #8/#9/#12 时发现 5 个 SHA 内容完全相同）。**"PR 已合并"这个状态本身具有欺骗性**，必须额外确认合并的目标分支。

### 2. 迁移脚本版本号靠人工数序，容易撞车

`V1.2.16` 先后被两个互不知情的功能（Stripe Customer 建表 / 行为看板菜单）占用，如果不是这次手工抽取时发现，两边谁先合并、谁的迁移脚本先执行都会造成混乱。且该项目**没有接入 Flyway**，全靠文档约定"手动执行"，清单文档还曾经写错成"自动执行"（已在这次更正）。

### 3. "先改代码"和"先改数据"的时序颠倒，会双向产生同一类事故

无论是"数据格式升级了、代码没跟上"（DataScope、Ecoinvent_v 3.10.0）还是"代码切了 fail-closed、数据没跟上"（本次特意规避的 7 条 ALL 套餐场景），本质都是**读写两端对同一份数据的格式假设不同步**。这次能提前规避后者，是因为吃过前者的亏之后才想到要反向检查。

### 4. 静默失败（`check=False` / try-except-pass 类模式）反复出现，且这次真实造成了业务影响

feedback 巡检的 bug 是本次最典型的例子：子进程失败被吞掉，业务状态（"已见"）却照常推进，导致问题存在了至少一周（6/26 PATH 修复之后）都没被发现，直到用户主动观察到"issue 很多但通知是 0"这个反常现象才被追查出来。这和 memory 里"静默 catch 是 bug 温床"记录的模式完全一致，说明**这类反模式不是一次性教训，需要更系统性的排查方式**（比如对所有 `check=False`/`subprocess.run` 不检查返回码的地方做一次专项扫描）。

### 5. 生产验证依赖临时性凭证（`KS_TOKEN` 约 2 小时过期），中途多次断线

排查生产日志时反复遇到 `KS_TOKEN` 过期（401），打断排查节奏，需要用户手动去浏览器复制。

## 可以改进的措施

1. **每次"合并到 main"的确认动作，显式检查 base branch**：不能只看 `state: MERGED`，必须核对 `baseRefName == main`。可以考虑写一个小脚本定期扫描"合并到非 main 分支、且该分支长期没有再合并回 main"的 PR，主动预警而不是靠人工翻。
2. **迁移脚本版本号改成基于时间戳而不是人工数序**（如 `V20260703103000__xxx.sql`），从根源上消除撞车可能；同时把"这个项目没有 Flyway、必须手动执行"这条写进项目级 README 或 CLAUDE.md，而不是散落在各个清单文档里、容易被复制传抄出错。
3. **涉及数据格式变更（不管是代码升级还是数据回填）时，固定问一遍双向问题**："如果反过来会怎样"——本次已经知道"代码升级、数据没跟上会崩"，这次能主动想到反过来查"数据升级、代码没跟上会不会崩"（避免了对 7 条 ALL 套餐的误伤），应该把这个检查动作固化成清单里的标准步骤，而不是靠临场想到。
4. **对项目里所有"静默失败"模式做一次专项扫描**：搜索 `check=False`、裸 `except: pass`、`subprocess.run(...)` 后不检查 `returncode` 的地方，逐个评估"失败了会不会被后续代码误当成成功继续推进业务状态"，尤其是巡检/自动化类脚本——这类脚本没有实时用户在盯着返回结果，最容易在静默失败后被忽视很久。
5. **`KS_TOKEN` 短期有效的问题**，如果生产日志排查是高频需求，值得跟平台方确认有没有更长效的服务账号 token 方案，减少人工在"排查中途去浏览器复制 token"这类打断性操作上的损耗。
